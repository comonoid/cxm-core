# CXM — платформа психологов: топологии A и B (ОБЕ)

> **Статус:** план (2026-07-03); отмечать `- [x]`. Self-contained для исполнения с чистого
> контекста (конвенции исполнения — как в `cxm-concept-upgrade-plan.md`: typecheck на модуль,
> стражи layering+neutrality, refl-тесты, live-smoke через unix-сокет, отметки в этом файле).
> **Решение пользователя:** нужны ОБА варианта — A (флот личных инстансов + community-инстанс)
> и B (один multi-tenant инстанс со многими психологами).
> **Контекст:** ядро контента/опыта готово (cxm-social-plan §1–10 реализованы); не хватает
> tenant-честности ядра (для B), control plane (для A), морды, гигиены (см. gap-анализ ниже).

## Зафиксированные решения (не переигрывать)

Д1. **Одно ядро — обе топологии.** Tenant-честность встраивается в ядро; A/B = конфиг деплоя:
    инстанс с 1 tenant ≡ сегодняшний single-operator (вырождение, не режим); площадка B =
    инстанс с N tenants; топология A = флот (n=1)-инстансов + один B-инстанс-«площадка».
Д2. **Две зоны данных.**
    - **ПУБЛИЧНАЯ зона** (кросс-tenant, правят политики canAccess/canList): Resource,
      ResourceLink, Mention, social-события, follow-рёбра. Лента/тред/витрина площадки B
      читают ЧЕРЕЗ tenants — контент психолога публикуется в ЕГО tenant, виден всем по
      политикам узла.
    - **КАБИНЕТНАЯ зона** (строго tenant-scoped по принципалу): Subject, Identity, Episode,
      Appointment, Payment, Account, Knowledge, Expectation/Promise, Outbox, Users,
      Assignments, IntegrationTokens. Психолог А НЕ видит клиентов Б.
Д3. **Принципал → tenant.** JWT sub = login → `User` → `uTenant`. Операторский dispatch
    ПОЛУЧАЕТ принципала (сейчас `dispatch h req` без него — ripple) и все кабинетные
    хендлеры фильтруют/пишут tenant принципала, НЕ `defaultTenant`. Platform-роль
    (`admin` со scope `"*"`) видит кросс-tenant (админ площадки).
Д4. **/v1 и tenants.** IntegrationToken УЖЕ несёт `itkTenant` → ingest/публикации через токен
    падают в его tenant. Публичные чтения /v1 (feed/thread/showcase/conversation/progress) —
    кросс-tenant (публичная зона).
Д5. **Психолог в B = User + Subject.** Оператор своего tenant + автор контента. Связка:
    `User += uSubject : Maybe ℕ` (Tier-1) — авторство контента оператора; заполняется при
    онбординге.
Д6. **A-синдикация — через уже готовый /v1.** Личный инстанс психолога постит на
    площадку-инстанс СВОИМ integration-токеном площадки (webhook/worker уже умеют доставку;
    паблиш — обычный POST /v1/publish). Identity-link между инстансами — явная закладка
    (выключен по умолчанию, §7.6).
Д7. **Деньги.** A: свой YooKassa на личный инстанс (работает сегодня). B: продажи контента
    через аккаунт площадки + fulfilment-as-data; **выплаты психологам (split/payout) —
    отдельная поздняя фаза** (в ядре появится только учёт долей — Account per tenant уже есть).
Д8. **RBAC-полиси — модуль над готовым authz-хуком** (route→требуемая роль/scope из
    scopedRolesOf), НЕ правка ядра маршрутизации. nginx-JWT остаётся внешним периметром.
