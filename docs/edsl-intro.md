# Как устроен наш EDSL — и почему им легко пользоваться

## Одна идея

Весь EDSL держится на одном приёме: **мы ничего не исполняем сразу — мы строим описание, а смысл
ему придаёт интерпретатор.** Описаний три вида, интерпретаторов у каждого минимум два (чистый
«эталонный» и постгресовый). Отсюда всё остальное.

## Три слоя

**1. Схема — описание данных** (`Agdelte.Storage.Schema`, схемы — `Cxm.Wire`). Таблица = список
колонок с типами (`mkCol "email" CStr`, `idxCol "tenant" (CFK "tenant")`, `CMaybe` = nullable).
Из одной схемы *выводятся*: типизированная строка `Row s` (тип вычисляется из схемы), WAL-кодек,
`CREATE TABLE` с индексами, `INSERT/UPSERT/DELETE/SELECT` (`Storage.SQL`), обратный декод
JSON→`Row` (`Storage.JsonRow`), имена и ТИПЫ индексных колонок для Exec (`Tables.idxCols/idxColTys`).
Написал схему — получил все интерпретации; разъехаться им не из чего — источник один.

**2. Верболы — описание команды** (`Cxm.Store.Verbs` + freer-ядро `Agdelte.Storage.Free`).
Команда — программа из атомов: `lockRoot`/`lockKey` (конкурентная дисциплина), `get/byIx/byCol/scan`,
`put/del`, `fresh`. Каждый вербол — конструктор данных; программа — дерево «вербол + продолжение».
Интерпретаторов три: чистый тест-хэндлер (`VerbsTest.handlerP` — refl-доказательства без БД,
он же чекер lock-дисциплины), нативный над `Base` (`VerbsBase`, runtime) и постгресовый
(`Cxm.Store.Pg.exec`: вербол → один SQL-стейтмент → `runTxPg` = BEGIN…COMMIT/ROLLBACK).

**3. Запросы — описание агрегата** (`Storage.Query`). Для тяжёлых выборок: терм
`countWhere "knowledge" (eqN "subject" 5 ∷ eqN "tenant" 2 ∷ [])` с чистым фолдом (эталон) и
компилятором в `SELECT COUNT(*)…`. Растёт по мере горячих путей — не наперёд.

Плюс **миграции — тоже термы** (`Storage.Migration`): `up`-SQL / `down`-SQL / чистая модель, и
сторож `migrate history [] ≡ cxmSchemas` — изменил схему без миграции ⇒ не компилируется.

## Почему это несложно применять

**Команда выглядит как обычный монадический код** — та же нотация, что в старом `Txn`:

```agda
bindIdentityV sub ch ext v ten now =
  lockRoot tcSubject sub >>T
  require tcSubject sub NotFound >>=T λ s →
  guardT (sTenant s ≡ᵇ ten) NotFound >>T
  byCol tcIdentity "external_id" ext >>=T λ hits →
  …find-or-create…
```

Никакой видимой «магии EDSL»: пишешь бизнес-логику как писал. То, что это *дерево данных*,
замечаешь в одном месте — при выборе интерпретатора.

**Ни строчки SQL руками.** Весь SQL эмитится из схемы и термов, каждая эмиссия пришпилена
`refl`-тестом (точная строка проверяется при typecheck) и live-диффом (`pg-diff`: native ≡ PG
на настоящем постгресе).

**Ошибки ловятся до базы.** Опечатка в колонке запроса — compile error (`T (hasNatCol …)`-гард).
Запись мимо lock-дисциплины, byCol/byIx мимо реестра — детерминированное падение чистого теста.
Тесты команд гоняются без БД вообще; часть — вообще `refl` (ноль рантайма).

**Расширять — по конструктору.** Новая таблица = строка в схеме + пункт реестра (пины поймают
пропуски; отдельно осознанно решить `appendOnly`/`queueTable`). Новый вербол = конструктор + по
клаузе в хэндлерах (компилятор укажет все места). Новый запрос = терм.

## Правила программиста серверной части (весь онбординг)

1. Команды — в Tx-нотации; **атом = один терм**, отданный в один `runCxmTx` (коммиты невыразимы,
   IO-верболов нет — «письмо из транзакции» = outbox-строка в том же атоме).
2. Первым делом `lockRoot` агрегата(ов) (мульти — через `lockRoots`, порядок сортирует комбинатор);
   создаёшь-если-нет → `lockKey` advisory. Свой залоченный агрегат стабилен; чужое — снапшот-взгляд.
3. `fresh` немедленно материализуется `put`'ом того же id. `byIx` — множество (порядок не
   специфицирован). SQL руками не писать; горячая выборка → query-терм.
4. Тесты — чистым хэндлером, без БД; перед релизом транслятора — `pg-diff` на scratch-PG
   (`scripts/pg-scratch.sh`).

Остальное принуждается типами, стражами и пинами, а не код-ревью.

## Конкурентность (READ COMMITTED, без сюрпризов)

Коммиты/анлоки/вложенность невыразимы: bracket — в интерпретаторе, локи — строгий 2PL структурно
(вербола unlock нет), `abort` → ROLLBACK целиком. Программист думает о трёх вещах: *что есть атом*,
*какие агрегаты трогаю*, *на что опираюсь*. Вложенные транзакции зарезервированы (`sub`-узел +
SAVEPOINT — см. pg-store-plan).

## Куда это растёт

PHOAS/selective-слой (после катовера): лямбды-биндеры над непрозрачными Var ⇒ всё дерево команды —
данные ⇒ один SQL-батч на команду, статическая проверка lock-set/fresh-пар/advisory-ключей.
Лестница: applicative ⊂ selective ⊂ монада — всё над тем же `Req`.
