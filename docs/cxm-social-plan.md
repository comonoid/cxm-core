# CXM × соцсеть (LJ-типа) — обзор сочетания, закладки, минимум

> **Статус:** обзор зафиксирован; минимум исполняется (2026-07-02), отмечать `- [x]`.
> **Рамка (уточнение пользователя):** нужна сеть типа LiveJournal, но (а) технически аккуратнее
> (живые обновления, дерево тредов перестраивается само) и (б) НЕ «LJ + цена на контент», а
> СОЧЕТАНИЕ с CRM/CXM: обозревать участника, выстраивать отношения, вести опыт.

---

## 1. Обзор: почему это не «LJ с ценником» — сочетание по конструктам ядра

Ключ: **социальная активность пишется в ТОТ ЖЕ append-only лог опыта и тот же граф субъектов**,
что и «клиентская» — поэтому CXM-машинерия (знание, отношения, решения) работает над ней даром.

| LJ-понятие | CXM-воплощение | Что даёт СВЕРХ LJ |
|---|---|---|
| Пост | `Resource` (узел дерева, `rAuthor`, `rVisibility`) + событие `Publish` | контент = адресуемый актив: на узел можно продать доступ, дерево = разделы/циклы |
| Коммент / тред | дочерний `Resource` (`rParent`) — дерево уже генерик | `threadOf` — чистая [ПР] проекция: дерево ПЕРЕСТРАИВАЕТСЯ из снапшота by construction |
| Лайк/реакция | `Reaction` peer-событие (`eeCounterpart` = автор, sentiment/peak-аннотации!) | реакция — факт ОПЫТА обеих сторон: пики памяти, сентимент-траектория автора |
| Френды | `follow`-ребро (+ `membership` для кругов/сообществ) | одна графовая механика с participation/decision_consult — «отношения» едины |
| Рейтинг/карма | `contributionOf` / `coSupportShare` — публичная свёртка вклада (слой IX) | репутация ВЫВОДИМА и пересчитываема; потеря статуса ловится как негативный пик |
| Платный доступ | `Entitlement TResource` на УЗЕЛ (пост/раздел/всё дерево), `Offering`-пакеты | + фьючерсы обещаний: ПЕРЕДАВАЕМАЯ подписка, залог по счёту доверия |
| Подписка на автора | follow + (опц.) entitlement на корневой узел автора | «обещал серию постов» = `Promise` автора: срыв виден, reliability двусторонний |
| Профиль автора | `subjectProfile` + knownAbout + reliability + их лента + их реакции | **«обозреть участника»** — карточка отношений оператору И self-facing прогресс самому |
| — (нет в LJ) | стиль текстов → гипотезы (метапрограммы/убедитель, Ч.4 AI-edge, INFERRED<1) | сеть НАПОЛНЯЕТ знание; конфликт «заявленное vs выявленное» — сигнал |
| — (нет в LJ) | Decision API над peer-событиями | «выстроить отношения»: кого поддержать, recovery на провал, co-support-подсказки |

Итого: сочетание = **сеть как ещё один канал опыта** (Community) + **контент как актив** +
**участники как субъекты полного профиля**. Оператор-психолог (или любая вертикаль) видит
клиента и как читателя/автора — одна лента, один субъект, одни отношения.

## 2. «Технически аккуратнее»: где живут живые обновления и самоперестройка

- **Дерево само перестраивается — на двух уровнях.** Сервер: `threadOf`/`feedOf` — чистые
  проекции из снапшота (никакого денормализованного состояния — нечему протухать). Фронт:
  agdelte reactive islands — дерево = `foreachKeyed` по проекции, ключ несёт содержимое ⇒
  узлы перерисовываются сами при изменении модели (тот же механизм, что консоль).
- **Живые обновления — закладки по нарастающей:** (1) минимум — poll `/v1/me/feed`;
  (2) push в браузер — **WebSocket уже в стеке** (`Agdelte.Http.serveWithWs`, warp/wai-websockets)
  — edge-инкремент виджета (cloud-Фаза 3); (3) внешним системам — исходящие вебхуки (готовы, блок D).
- **Редактирование поста** — закладка: `rUpdatedAt`/`updateResource` (Tier-1 добавка) — вне минимума.

## 3. Зафиксированные решения (не переигрывать)

1. Пост/коммент = `Resource` (`rAuthor : Maybe ℕ`, Tier-1); отдельной сущности «пост» нет;
   тред = `rParent`-дерево.
2. Политика доступа = данные-строки + одна тотальная `canAccess` (nothing/"public" | "followers"
   | "entitled"); неизвестная ⇒ deny; автор видит своё. Расширения («circle:<id>» через
   membership) — новые строки, не enum.
3. Лента и тред = чистые [ПР] (`feedOf`, `threadOf`); материализация — только при доказанной
   нужде.
4. `Publish`/`Reaction` = типы ExperienceEvent; публикация журналируется в одной Txn с ресурсом;
   реакция = обычный peer-ingest `/v1/events` (ничего нового).
5. Подписка = `addEdge follow`; обе стороны на `/v1` — через identity-bridge.
6. Монетизация — готовые швы: `grantEntitlement TResource` (+ POST /entitlements в минимуме);
   онлайн-оплата узла = существующий payments-поток (вне минимума).
7. Новый модуль `Cxm.Social` — L5, со спекой (MODULES.md) и зелёным check-layering.
8. **Вне минимума (закладки, зафиксировать в MODULES.md):** rUpdatedAt/updateResource;
   WS-push виджета; круги (membership-политика "circle:<id>"); AI-извлечение из текстов (Ч.4);
   передаваемая подписка (уже возможна фьючерсами — нужен только сценарий).

## 5. Принципы совмещения CRM/CXM × соцсеть (пример: платформа психологов типа b17)

> Вертикальные слова — ТОЛЬКО здесь (план), не в ядре (страж нейтральности).

1. **Один субъект во всех ролях.** Читатель статьи = автор вопроса = клиент терапии — identity-
   bridge сшивает; весь ДО-клиентский контентный опыт уже в ленте субъекта. (Реализовано: /v1
   ingest + provisional+merge.)
2. **Контент = касание опыта, не витрина.** Пост/вопрос/реакция — ExperienceEvent с sentiment/
   peak → путь «читал → спросил → записался» читается как макромодель решения. (Publish/Reaction
   + eventTypeSequence.)
3. **Сеть наполняет знание — эпистемически честно.** Тексты → ГИПОТЕЗЫ (INFERRED<1, evidence=
   пост) о JTBD/метапрограммах/убедителе; публичная персона ≠ приватная (ниже conf, корроборация).
   Специалист входит в первую сессию не вслепую. (Конверт готов; извлечение — Ч.4 edge.)
4. **Отношения = граф + экономика доверия, не «френды».** follow/membership/participation —
   одна механика; обещания двусторонние («отвечу за день» — Promise специалиста; no-show клиента
   виден); reliability — обе стороны; convincer «N касаний» набирается КОНТЕНТОМ до продажи.
5. **Монетизация = грант доступа, не ценник.** Entitlement на узел/раздел/дерево, Offering-
   пакеты, ПЕРЕДАВАЕМЫЕ обещания (подарить подписку/сессию), залог от счёта доверия. (Всё в ядре.)
6. **Сообщество — измеряемый со-производитель опыта.** Peer-ответы на вопросы (как «вопросы
   психологам»), но: co-support vs наша SLA, вклад/репутация ВЫВОДИМЫ, потеря статуса =
   негативный пик под recovery. (contributionOf/coSupportShare/statusDropPeaks.)
7. **Decision API поверх сети.** Негативный пик в комментах → recovery; дозревшая JTBD-гипотеза →
   предложить «осмотреться»; низкая уверенность → explore-касание контентом. Next-best-action
   работает на читателях, не только клиентах.
8. **Два контура приватности жёстко разделены.** Публичная репутация (слой IX) ≠ приватное
   терапевтическое знание (Knowledge, крипто-шред политика); изоляция инстанса; политики
   доступа - данные (canAccess).
9. **Платформенная развилка (ОТКРЫТА, решить при cloud-Фазе 4):** b17 — единая площадка; у нас
   изоляция instance-per-специалист. Сочетание: общая контент-площадка = ОТДЕЛЬНЫЙ community-
   инстанс, терапия — в личных; связь идентичностей МЕЖДУ инстансами — выключена по умолчанию
   (§7.6), включается явной закладкой. Не решать мимоходом.
10. **Технически аккуратнее LJ:** headless-проекции (feedOf/threadOf — нечему протухать) +
    реактивные острова на фронте (перестройка по ключам) + WS-push как закладка (serveWithWs в
    стеке) + вебхуки наружу (готовы).

## 6. Экономика продаваемого: контент × обещания (одна рамка)

**Всё продаваемое — ровно два рода товара, оба уже first-class в ядре:**
1. **Товар-ДОСТУП** (состояние «у тебя есть»): пост, документация, видео, раздел, членство →
   `Entitlement` (TResource на узел/ДЕРЕВО — грант наследуется вниз по rParent, entitledUpᵇ;
   TOffering; TMembership) с validTo (подписка) или бессрочно.
2. **Товар-ОБЯЗАТЕЛЬСТВО** (состояние «тебе должны»): сессия, «отвечу за 24ч», серия постов,
   разбор → `Promise` (direction Ours, deadline, transferable, collateral) с клиринговым
   журналом list/transfer/settle/default.

**Связка через каталог:** `Offering` = что продаётся (цена/метаданные); успех оплаты
(`markPaymentSucceeded`) выдаёт грант. **Закладка (fulfilment-as-data):** маппинг «offering →
какие гранты И/ИЛИ обещания создать при успехе» — данные оффера (oMetadata), интерпретатор —
малый инкремент; сейчас составные исполнения собирает пак/оператор (grantEntitlement +
createPromiseDirected в одной Txn).
**Отличие родов:** доступ потребляется без нашего действия (canAccess); обещание требует
исполнения и попадает в reliability/доверие. «Подарить купленное» сейчас = передаваемое
ОБЕЩАНИЕ доступа (фьючерс); переносимый Entitlement (holder) — закладка, вводить только со
сценарием.

## 7. S7 — видимость каталога ≠ читаемость содержимого (нюанс пользователя)

**Решение:** у узла ДВЕ политики: `rVisibility` (кто ЧИТАЕТ payload) и `rListing : Maybe String`
(кто ВИДИТ существование узла в ленте/треде/каталоге; тот же словарь строк; nothing = как
rVisibility — т.е. по умолчанию нечитаемое невидимо, витрина включается ЯВНО listing="public").
Проекции возвращают `ContentView {locked, resource}`: listed∧¬readable ⇒ locked=true, payload
в JSON вырезается (тизер: заголовочные метаданные без содержимого). Поддерево листится дальше,
если сам узел listed (свои политики у детей); нелистуемый узел прячет ветку, как раньше.
- [x] `rListing` (Tier-1 CMaybe CStr, конец схемы) + ripple (createResource/publishResource/
      call-sites/Wire/тесты).
- [x] Social: `canList` + `ContentView`; `feedOf`/`threadOf` → views с locked; refl-тесты
      (витрина: locked-тизер в ленте у не-покупателя; скрытый черновик не листится).
- [x] Api: параметр `listing` у публикаций; тизер-JSON (payload="" + "locked":true).
- [x] Smoke: платный раздел с listing=public — не-покупатель ВИДИТ оглавление с locked, купил →
      payload раскрылся; узел без listing — невидим (как раньше).

## 8. S8 — структурирование контента: витрина/промо/рейтинг (вопрос пользователя)

**Задача:** индексировать куски больших видео отдельно; рекламировать на Главной (это ПРОДАЖА
рекламного места); вести рейтинг индексатором. **Структуры:**
1. **Фрагмент = обычный узел** `kind=clip`, payload `{"src":<rid>,"from":s,"to":s}` — кусок
   ссылается на большое видео ДАННЫМИ; публикуется/продаётся/анонсируется как любой узел
   (ничего нового в ядре).
2. **`ResourceLink`** (НОВАЯ сущность, реализована): кураторский ГРАФ ссылок между узлами —
   отличен от дерева composition (rParent). Поля: from(витрина)/to(контент)/kind-строка
   (pin/promo/related — данные)/rank(кураторский порядок)/validFrom/validTo/tenant.
   **Продажа рекламного места = link с validTo** (оплаченный срок): fulfilment оплаты зовёт
   `linkResource`; истечение НЕ требует воркера — витрина-проекция фильтрует по окну.
3. **`showcaseViews`** (проекция, реализована): live-окно ссылок витрины → ранг-asc →
   каждый target в listing/locked-семантике S7 (тизер платного прямо на Главной).
4. **Рейтинг:** кураторский = rank ссылки (сейчас); ВЫВОДИМЫЙ = проекция-счётчик по
   Reaction-событиям с ЭКСТРАКТОРОМ target'а параметром (ядро payload не парсит; Api подаёт
   экстрактор через JSON-FFI) — ЗАКЛАДКА, вводить со сценарием.
- [x] ResourceLink: record+Wire+Base(@/x)+Codec+Interface+`linkResource`/`unlinkResource`.
- [x] `showcaseViews` в Cxm.Social.
- [x] HTTP: POST/GET `/resource-links`, `/v1/showcase`; refl+smoke (ранги, истёкший слот исчез,
      locked-тизер в витрине).

## 9. Контент = markdown (docsify-стиль) — решение по формату

**Решение:** payload узла = markdown-текст + конвенция полей
`{"format":"markdown","title":…, "body":…}` (opaque для ядра — §8.1: ядро НЕ парсит/не
рендерит). Рендер — ЦЕЛИКОМ фронт (cloud-Фаза 3, виджеты): docsify-подход (клиентский рендер
без сборки) поверх `/v1/thread|feed|showcase` вместо .md-файлов; плагины = mermaid, KaTeX/
MathJax и пр. — конфиг виджета, не ядро. WYSIWYG НЕ нужен на старте: textarea с markdown
(+preview тем же рендером). Ядру НИЧЕГО не требуется — формат уже opaque-совместим.

## 10. Разговоры от любой сущности: формат, кодирование, хранение (обсуждено 2026-07-02)

**Развилки зафиксированы (F1–F4):** комменты = Resource (вся policy/teaser/модерация-машинерия
наследуется); якорь ДЕНОРМАЛИЗОВАН на каждом узле разговора; адресаты = child-таблица Mention
(ordinal, 0 = основной, дублируется в eeCounterpart peer-события); аудитория разговора =
аудитория якоря (резолвер по kind) ∧ собственные политики узлов.

**Формат:** `Resource += rAnchorKind : String` ("" = не разговор; "resource"|"appointment"|
"promise"|"entitlement"|"subject"|"episode"|…) + `rAnchorId : ℕ` (0-sentinel, idxCol —
ИНДЕКСИРОВАН; выборка всего разговора = один индекс-скан + kind-фильтр). Дерево — adjacency
`rParent`; порядок хранения ЛИНЕЕН (append), деревянность логическая.
`Mention {mId, mResource(idx), mSubject(idx!), mOrd, mTenant}` — «ответ нескольким, кому-то
первому»; inbox упоминаний = скан по mSubject; уведомления через готовый Outbox.

**Хранение (синтез обсуждения):**
1. **Истина = мелкозернистые строки + наш op-log** (каждый коммент пишется РАЗ; O(n²)-ловушка
   есть только у «блоба-как-истины»; в Postgres-будущем skinny rows inline — TOAST не включается).
2. **Чтение = якорный индекс + `since`-курсор + пагинация** («клиентская сборка WAL»,
   серверная фильтрация canList/locked ДО отдачи).
3. **ЗАКЛАДКА: линейный page-кэш** (не истина, [ПР]-материализация): `ConversationPage {anchor,
   pageNo, fromId, toId, body}` — запечатанные страницы ИММУТАБЕЛЬНЫ (⇒ ETag/CDN вечные),
   перезаписывается только хвост; никакого B-tree (только «хвост дорос → запечатать»); формат
   body — НАШ производный артефакт (парсить можно, §8.1 не нарушается) с policy-метаданными
   узлов — фильтрация скрытого ПРИ ОТДАЧЕ; модерация = пересборка страницы. Включать по
   профилированию.
4. **Подписка естественна из лога:** pull `since` / push WS / вебхуки — один контракт «всё
   после id N», три транспорта. История(страницы)/настоящее(хвост)/будущее(подписка) — одна ось id.

**Минимум к реализации:** anchor-поля + Mention + `requireAnchor` + команды `commentOn`
(якорь+parent+адресаты+peer-событие одной Txn) + `/v1/comment`, `/v1/conversation {anchor_kind,
anchor_id, since, limit}` (+ mentions в ответе), тесты + smoke.

**Уточнение §10.3 (по-ветковые права × страницы, обсуждено):** вместо фильтрации-при-отдаче
(парсинг чанков на горячем пути + потеря CDN из-за auth-вариативности) принято:
**страницы содержат ТОЛЬКО audience-инвариантные (публичные) узлы**; приватные/followers/
entitled-ветки НИКОГДА не входят в общий кэш и едут пер-зрительской дельтой `since` (фильтр на
строках истины, без чанков). Клиент мержит «публичный скелет страниц + персональная дельта»
(та же id/parent-сборка). Выгоды: ноль serve-парсинга, страницы иммутабельны И CDN-безопасны,
приватное физически отсутствует в разделяемом кэше. Fallback-приём для пересборки страниц —
length-prefixed фреймы с policy-байтом (skip без парсинга содержимого).

**РЕШЕНИЕ §10.5 (права × чанки — вариант пользователя, ПРИНЯТ; обобщает §10.3):**
дерево разговора распадается на НЕСКОЛЬКО линейных чанк-серий («несколько WAL'ов»):
**поток = максимальное поддерево с одним policy-корнем**. Редукция реализации:
